# 中栏图片加载优化设计

- 日期：2026-03-06
- 状态：已确认（Approved）
- 范围：
  - `src/features/articles/ArticleList.tsx`
  - `src/features/articles/ArticleList.test.tsx`
  - 仅限 reader 中栏文章卡片预览图加载策略

## 1. 背景与目标

当前 reader 中栏卡片图的加载策略是：先为当前列表里的所有候选图片执行 `new Image()` 预加载，只有预加载成功后才真正渲染 `<img>`。这一策略避免了失败图片先出现再消失的闪动，但也带来新的体感问题：首屏图片需要与屏外图片共享请求带宽，导致中栏初始可见图片出现偏慢。

本次优化目标：

1. 让首屏可见文章卡片图片更快出现。
2. 改为滚动按需加载，而不是对当前列表候选图全量并发预加载。
3. 不引入占位图、skeleton 或“先出现后消失”的闪动。
4. 在按需加载基础上，仅额外预取下方 `0.5` 屏范围，兼顾滚动体验与请求量。

## 2. 已确认约束

1. 不接受占位图；占位导致图片失败后消失时会产生明显闪动。
2. 保持“预加载成功后才渲染图片”的契约，不退回到直接渲染 `<img>` 的模式。
3. 允许只优先加载可视区与少量下一屏图片，不要求远处图片尽早全量预取。
4. 预取范围固定为下方 `0.5` 屏，不采用 `1` 屏或 `2` 屏的更激进策略。
5. 本次不引入虚拟列表，不拆 reader snapshot 接口，也不改后端数据结构。

## 4. 方案比较与选型

### 方案 A（采纳）：可视区驱动预加载 + 下方 `0.5` 屏预取 + 并发限流

- 形态：
  - 保留候选图提取逻辑与 `ready / failed` 渲染契约。
  - 使用 `IntersectionObserver` 以中栏滚动容器为 `root`，仅激活可视区及下方 `0.5` 屏内的图片 key。
  - 通过小型队列限制同时进行的 `Image()` 预加载数量。
- 优点：
  - 首屏图片不会被屏外大量候选图抢带宽。
  - 滚动时仍有前瞻预取，减少“滚到才开始加载”的等待感。
  - 改动集中在 `ArticleList`，不影响 snapshot / store 契约。
- 缺点：
  - 需要增加可视区观测与预加载调度逻辑。
  - 测试需要补充 observer 与限流场景。

### 方案 B：首屏优先预加载，其余候选图随后继续全量预加载

- 优点：改动更小，能改善“第一眼太慢”。
- 缺点：屏外图片仍会在第二阶段继续抢网络资源，对滚动中的收益有限，无法从根本上解决长列表全量预取问题。

### 方案 C：移除 JS 预加载，只保留原生 `<img loading="lazy">`

- 优点：实现最简单，依赖浏览器默认懒加载。
- 缺点：无法满足“不占位、不闪动、失败别先出现再消失”的约束，也会破坏现有测试契约。

结论：采用方案 A。

## 5. 架构与组件设计

### 5.1 总体思路

1. 保留 `previewImageByArticleId` 作为候选图来源，但不再直接对全部候选图发起预加载。
2. 新增一层“激活态”集合 `activePreviewImageKeys`，只让已进入可视区或预取区的图片进入加载队列。
3. 继续沿用 `previewImageStatuses` 作为最终渲染依据：只有 `ready` 才显示图片，`failed` 永不显示。

### 5.2 观察与激活边界

1. 以中栏滚动容器作为 `IntersectionObserver.root`。
2. 观察目标挂在文章卡片根节点，而不是图片节点本身。
   - 原因：图片在 `ready` 之前并未渲染，无法稳定作为 observer target。
   - 卡片节点始终存在，更适合在图片尚未出现时提前激活加载。
3. `rootMargin` 向下扩展 `50%` 容器高度，对应“预取下方 `0.5` 屏”。

### 5.3 加载队列

1. 预加载从“遍历全部候选图”改为“只消费激活集合里的未处理 key”。
2. 队列并发限制为 `2`，优先保证首屏图片尽快完成。
3. `loading / ready / failed` 状态的 key 不重复入队。

## 6. 数据流与时序

### 6.1 初次渲染

1. 先渲染文字信息，不渲染未就绪图片。
2. observer 计算当前可视区与下方 `0.5` 屏中的卡片。
3. 这些卡片对应的图片 key 加入 `activePreviewImageKeys`。
4. 队列以至多 `2` 个并发执行 `Image()` 预加载。
5. 图片完成加载后，将对应 key 标记为 `ready`，再触发该卡片渲染 `<img>`。

### 6.2 滚动过程

1. 新进入可视区 / 预取区的卡片被加入激活集合。
2. 已 `ready` 或 `failed` 的 key 不重复请求。
3. 已离开视区较远的卡片不需要主动降级为未激活，但后续不会继续把未激活的屏外图片塞入队列。

### 6.3 视图切换与快照刷新

1. 当 `selectedView` 切换、snapshot 重新加载或列表集合变化时，移除已不存在 key 的图片状态。
2. observer 重新绑定当前列表项，激活集合按新列表重新计算。
3. 已不在候选集合中的 key 不保留，避免状态表无限增长。

## 7. 错误处理与性能护栏

### 7.1 错误处理

1. 预加载 `onload` 后标记为 `ready`。
2. 预加载 `onerror` 后标记为 `failed`。
3. 已经渲染的 `<img>` 仍保留 `onError` 兜底，覆盖“预加载成功但真实渲染失败”的边缘场景。

### 7.2 性能护栏

1. 不再为整个列表的所有候选图一次性创建 `Image()`。
2. observer 只在文章集合变化时重建，不跟随每次图片状态更新重建。
3. 队列运行态以 `ref` 管理，减少因状态联动造成的 effect 级联重跑。
4. 不引入虚拟列表，避免本次问题被扩大为布局与滚动体系重构。

## 8. 测试与验收标准

### 8.1 需要保留的现有契约

1. 预加载成功前，不渲染图片。
2. 预加载失败后，图片保持隐藏。

### 8.2 需要新增的覆盖

1. 初次渲染时，仅为可视区及预取区内的图片创建预加载请求。
2. 屏外较远卡片在未进入激活区前，不应触发 `Image()`。
3. 滚动 / observer 回调后，新进入预取区的卡片才开始加载。
4. `failed` 的图片在重复进入视区后不应反复请求。
5. 切换列表或刷新 snapshot 后，旧 key 能被清理，避免状态残留。

### 8.3 验收标准

1. 首屏图片的出现时间相较现状明显提前。
2. 滚动到下一屏时，多数图片已在预取区阶段开始加载，不出现大面积集中等待。
3. 不出现占位闪动、失败后消失、布局跳变等回归。
4. 现有中栏图片显示测试与新增可视区加载测试全部通过。
