# RSS 源右键移动分类设计

- 日期：2026-03-06
- 状态：已确认（Approved）
- 范围：
  - `src/features/feeds/FeedList.tsx`
  - `src/features/feeds/FeedList.test.tsx`
  - `docs/plans/2026-03-06-rss-feed-context-move-category-implementation-plan.md`

## 1. 背景与目标

当前阅读器左栏已经支持：

1. 在分类标题上通过右键菜单做 `编辑 / 上移 / 下移 / 删除`
2. 在 RSS 源上通过右键菜单做 `编辑 / AI摘要配置 / 翻译配置 / 启用停用 / 删除`
3. 在 `EditFeedDialog` 中修改 RSS 源所属分类

但“修改 RSS 源所属分类”仍然必须进入编辑弹窗，路径偏长，不适合高频整理操作。

本次需求目标：

1. 在 RSS 源右键菜单中新增 `移动到分类` 子菜单
2. 子菜单直接列出现有分类与 `未分类`
3. 点击目标后立即完成移动，不弹确认框，不打开额外弹窗
4. 当前所属分类在子菜单中可见但禁用，作为当前位置提示
5. 不在该菜单中承担新建分类能力，避免把右键菜单做成复杂输入场景

## 2. 已确认约束

1. 交互形态采用：`移动到分类 > [分类列表]`
2. 子菜单只显示现有分类和 `未分类`
3. 不支持在该子菜单中创建新分类
4. 当前所属分类显示但禁用
5. 点击目标后立即执行移动
6. 排序遵循现有分类顺序，不额外引入排序规则

## 3. 相关经验与已知约束来源

  - 启发 2：`updateFeed` 已经承担分类变更后的快照同步职责，应复用而不是新增平行 store action。

## 4. 方案比较与选型

### 方案 A（采纳）：订阅源右键菜单新增 `移动到分类` 子菜单

- 形态：在 RSS 源右键菜单中插入 `ContextMenuSub`，直接列出可移动目标。
- 优点：
  - 最符合“右键直接移动”的目标。
  - 可以直接复用现有 `updateFeed(feedId, patch)`。
  - 不需要新增对话框或复杂输入状态。
- 缺点：
  - 分类过多时子菜单会变长，但当前信息架构仍可接受。

### 方案 B：右键菜单只提供 `移动到分类...`，点击后打开小弹窗

- 优点：
  - 后续容易扩展搜索、分页或创建分类。
- 缺点：
  - 比目标交互多一步。
  - 重新把简单动作拉回弹窗，不符合当前收敛方向。

### 方案 C：继续复用 `编辑` 弹窗，不新增右键移动入口

- 优点：
  - 改动最小。
- 缺点：
  - 不能解决当前整理效率问题。
  - 与需求不符。

结论：采用方案 A。

## 5. 信息架构与交互设计

## 5.1 RSS 源右键菜单结构

建议 RSS 源右键菜单调整为：

1. `编辑`
2. `移动到分类 >`
3. `AI摘要配置`
4. `翻译配置`
5. `启用 / 停用`
6. `删除`

其中 `移动到分类` 使用现有 `ContextMenuSub` 组件，不新增新的菜单基础设施。

## 5.2 `移动到分类` 子菜单内容

1. 普通分类按 `categoryMaster` 当前顺序展示
2. 子菜单最后补一个 `未分类`
3. 当前所属分类项显示但禁用
4. 若当前 feed 处于未分类，则 `未分类` 项禁用
5. 子菜单只承担移动动作，不提供重命名、创建或删除分类

## 5.3 成功与失败反馈

1. 移动到普通分类成功后，提示 `已移动到「分类名」`
2. 移动到 `未分类` 成功后，提示 `已移动到「未分类」`
3. 失败时复用 `mapApiErrorToUserMessage(err, 'update-feed')`
4. 不在提交前做本地乐观挪动，因此失败时无需额外回滚

## 6. 数据流与状态设计

## 6.1 设计原则

本次改动应保持“最小状态面”：

1. 不新增 store action
2. 不新增本地分类事实源
3. 不在 `FeedList` 中手工推导分类集合增删
4. 继续由服务端和 snapshot 决定最终结果

## 6.2 提交语义

从 RSS 源右键菜单发起移动时，直接复用 `updateFeed`：

1. 目标为普通分类：`updateFeed(feed.id, { categoryId: targetCategoryId })`
2. 目标为 `未分类`：`updateFeed(feed.id, { categoryId: null })`

不使用 `categoryName`，因为该菜单不支持新建分类。

## 6.3 同步策略

`src/store/appStore.ts` 已在 `updateFeed` 成功后执行：

1. `patchFeed(feedId, patch)`
2. 更新本地 feed 基础字段
3. `loadSnapshot({ view: get().selectedView })`

因此本次功能不需要新增同步链路，就能自动获得以下行为：

1. feed 进入新分类分组
2. 若原分类因此变空，服务端清理后会从左栏消失
3. 分类顺序继续与 snapshot 保持一致

## 6.4 禁用态规则

为避免无意义请求，子菜单项禁用规则如下：

1. `feed.categoryId === category.id` 时，对应普通分类项禁用
2. `feed.categoryId === null` 且命中 `未分类` 项时禁用
3. 禁用项仍显示，作为当前位置提示

## 7. 测试策略

## 7.1 UI 测试

在 `src/features/feeds/FeedList.test.tsx` 中补充以下覆盖：

1. 订阅源右键菜单包含 `移动到分类`
2. 子菜单按分类顺序展示普通分类并包含 `未分类`
3. 当前所属分类项为禁用态
4. 点击普通分类后，patch body 为目标 `categoryId`
5. 点击 `未分类` 后，patch body 为 `categoryId: null`
6. 移动成功后，feed 出现在新分组
7. 原分组被挪空时，该分组从左栏消失
8. 当前未分类 feed 的 `未分类` 目标项为禁用态

## 7.2 回归测试

需要确保以下既有行为不受影响：

1. `编辑` 仍能打开 `EditFeedDialog`
2. `AI摘要配置` 与 `翻译配置` 菜单项仍可正常使用
3. `启用 / 停用` 与 `删除` 行为不变
4. 分类标题右键菜单行为不变

## 8. 验收标准

1. RSS 源右键菜单新增 `移动到分类` 子菜单
2. 子菜单只显示现有分类和 `未分类`
3. 当前所属分类在子菜单中可见但禁用
4. 点击目标后立即执行移动，不打开弹窗
5. 移动普通分类时请求体为对应 `categoryId`
6. 移动到 `未分类` 时请求体为 `categoryId: null`
7. 移动后左栏分组以 snapshot 结果为准更新
8. 原分类被挪空时不会保留空分类
9. `FeedList` 相关单元测试通过
10. `pnpm run lint` 通过

## 9. 非目标

1. 本期不在右键子菜单中支持创建新分类
2. 本期不引入分类搜索、分页或虚拟滚动
3. 本期不调整分类标题右键菜单的 `编辑 / 上移 / 下移 / 删除` 规则
4. 本期不修改服务端分类生命周期契约
