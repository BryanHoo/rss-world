# 阅读器右侧目录重设计

- 日期：2026-03-07
- 状态：已确认（Approved）
- 范围：
  - `src/features/articles/ArticleView.tsx`
  - `src/features/articles/ArticleOutlineRail.tsx`（建议在实现阶段重命名为更贴切的目录组件名）
  - `src/features/articles/articleOutline.ts`
  - `src/features/articles/ArticleView.outline.test.tsx`
  - `src/features/articles/ArticleOutlineRail.test.tsx`
  - `docs/plans/2026-03-07-reader-outline-panel-redesign-implementation-plan.md`
- 替代：本设计替代 `docs/plans/2026-03-07-reader-outline-rail-design.md` 中“窄轨道 + hover 展开卡片”的交互方向

## 1. 背景与目标

当前阅读器右栏已经存在一个“窄轨道 + hover 展开卡片”的目录实现，但这套交互仍然过于像一个悬浮控件，而不是正文阅读体验的一部分。用户希望把目录重新设计为：

1. 默认展开，但存在感低。
2. 不占用正文宽度，而是优先使用正文右侧到阅读区边缘的空白。
3. 目录可以点击跳转到文章章节。
4. 正文滚动后目录仍然停留在右上方，不会被正文滚走。
5. 当右侧空白不足时，目录自动变窄并截断，而不是压进正文。
6. 当文章过短时，即使存在标题结构，也不显示目录。

本次设计目标不是新增第四栏，而是在不破坏现有三栏布局的前提下，把目录变成“正文外侧的轻量辅助导航”。

## 2. 已确认方向与边界

### 2.1 已确认方向

1. 目录在桌面端默认展开。
2. 目录视觉上保持低存在感，不做强侧栏感。
3. 目录位于阅读区右上角，滚动后仍保持固定位置。
4. 目录宽度优先吃掉正文与阅读区右边缘之间的空白。
5. 右侧空白不足时，目录跟着变窄并对文字做单行截断。
6. 非桌面宽度下直接隐藏目录，不额外设计移动端目录交互。
7. 文章过短时隐藏目录；长度口径按最终渲染后的正文高度与当前视口高度比较。

### 2.2 不做内容

1. 不新增独立的第四列目录栏。
2. 不新增目录搜索、折叠、固定展开开关、拖拽调宽等控制能力。
3. 不把目录状态写入 `useAppStore` 或持久化设置。
4. 不针对移动端补充单独的目录抽屉、手势或弹层。
5. 不把文章页头的大标题自动并入正文目录。

## 3. 项目内经验与已知坑

### 3.1 相关总结

- `docs/summaries/2026-03-06-reader-resizable-three-column-layout.md`
  - 启发：右栏是三栏布局中的剩余空间列，目录必须作为附着层或浮层来实现，避免再次影响三栏宽度分配。
- `docs/summaries/2026-03-05-translation-preserve-html-structure.md`
  - 启发：翻译模式会改变正文最终渲染结果，因此目录必须基于最终 DOM 提取 heading，而不是从文章数据重建。
- `docs/summaries/2026-03-06-middle-column-image-loading.md`
  - 启发：与滚动和可视区联动的逻辑应保持局部状态与观察器驱动，避免高频写全局状态造成无意义重渲染。

### 3.2 已知风险

1. 如果目录继续挂在 `article-scroll-container` 的滚动内容内部，滚动时就有被带走的风险。
2. 如果把目录改造成正文旁的一列布局，会间接改变正文的宽度感知与居中关系。
3. 如果目录位置写死到视口边缘，而不对齐正文实际右边缘，会在不同栏宽下显得漂浮且不稳定。

## 4. 方案比较与选型

### 方案 A（采纳）：正文外侧独立浮层目录

- 做法：目录脱离正文滚动内容，作为阅读区内的独立附着层；通过测量正文容器和阅读区容器，动态计算目录的横向位置与可用宽度。
- 优点：
  1. 最贴合“占正文右侧空白、不挤正文”的目标。
  2. 滚动时天然不会被正文带走。
  3. 对现有三栏宽度模型侵入最小。
- 缺点：
  1. 需要增加一层测量与重算逻辑。
  2. 需要兼顾窗口缩放、栏宽拖拽、翻译视图切换后的重新定位。

### 方案 B：正文列 + 目录列壳层

- 做法：把右栏正文区改造成“正文列 + 目录列”的两列壳层，目录列使用 `sticky`。
- 优点：布局结构直观，目录列宽度表达简单。
- 缺点：会更明显地影响正文居中与宽度感知，且和现有三栏可拖拽布局更容易相互耦合。

### 方案 C：直接固定到视口右侧

- 做法：目录直接 `position: fixed` 到整个页面或视口右侧。
- 优点：实现最简单。
- 缺点：无法稳定对齐正文右边缘；在不同栏宽与不同窗口尺寸下容易显得“飘”。

结论：采用方案 A。

## 5. 布局与定位设计

### 5.1 总体结构

1. `article-scroll-container` 继续只负责正文滚动。
2. 正文内容区继续保持当前居中阅读宽度，不改变 `mx-auto` 与 `max-w-*` 的正文排版基线。
3. 目录改为 `ArticleView` 内的独立附着层，与滚动容器是同级或近同级关系，而不是正文流中的子内容。

### 5.2 垂直定位

1. 目录默认固定在阅读区右上角。
2. 顶部留固定安全间距，与设置按钮、浮动标题同属阅读辅助层。
3. 正文滚动只影响当前章节高亮，不影响目录本身的垂直位置。

### 5.3 横向定位与宽度

目录的位置与宽度由运行时测量得出：

1. 读取正文外层包裹容器的实际 `getBoundingClientRect()`。
2. 读取阅读区可视容器的实际 `getBoundingClientRect()`。
3. 目录左边界 = 正文右边缘 + 固定间距。
4. 目录最大宽度 = 阅读区右边缘 - 目录左边界 - 右侧安全边距。

实现阶段应把这段逻辑抽成纯 helper，并为其定义：

1. `minWidth`：目录可接受的最小宽度。
2. `maxWidth`：目录的舒适最大宽度。
3. `hideThreshold`：当右侧可用空白低于阈值时，目录直接隐藏。

### 5.4 宽度退化策略

1. 右侧空白足够时，目录以舒适宽度显示。
2. 右侧空白变小时，目录同步变窄。
3. 目录标题统一单行截断，不换行堆高。
4. 不允许目录反向侵入正文阅读宽度。

## 6. 内容与交互设计

### 6.1 目录内容来源

1. 仅提取正文里的 `h1/h2/h3`。
2. 不把文章页头标题自动并入目录。
3. 没有目录项时，整个目录入口完全不显示。

### 6.2 层级展示

1. `h1` 视觉层级最高。
2. `h2` 为常规层级。
3. `h3` 做轻度缩进与更弱强调。
4. 所有目录项都保持单行截断。

### 6.3 默认视觉与 hover 态

1. 目录默认常驻展开，但底色、边框、阴影、字重都保持克制。
2. 当前章节只做轻量高亮，不使用强饱和整块高亮。
3. hover 到目录区域时，只提升少量对比度或背景透明度，不再承担“展开/收起”的职责。
4. 不再以窄轨道作为主视觉，不再依赖 hover 才暴露完整目录。

### 6.4 点击跳转

1. 点击目录项后，正文滚动容器平滑滚动到对应 heading。
2. 落点保留固定顶部偏移，避免目标标题贴边顶死。
3. 点击时可立即切换当前高亮，随后由真实滚动位置进行校正。

### 6.5 目录内部可见性

1. 若目录项较多导致目录自身需要滚动，当前章节切换后应尽量保持当前项可见。
2. 目录的内部滚动只发生在目录浮层内部，不影响正文滚动。

## 7. 显示规则与退化策略

目录必须同时满足以下条件才显示：

1. 当前有选中的文章。
2. 当前为桌面宽度。
3. 正文最终渲染结果中存在 `h1/h2/h3`。
4. 阅读区右侧存在足够放置目录的空白。
5. 正文长度达到展示目录的最低阈值。

### 7.1 短文章隐藏规则

“文章过短”按最终渲染后的正文高度判断，而不是按原始文章字数、HTML 长度或目录项数量判断。

推荐规则：

1. 使用正文实际渲染高度与阅读区当前可视高度比较。
2. 当正文总高度不超过 `1.25` 屏时，隐藏目录。
3. 当正文明显超过一屏时，才显示目录。

这样可以自然覆盖：

1. 图片加载后正文变长。
2. 翻译视图切换后正文高度变化。
3. 窗口缩放后可视高度变化。

## 8. 状态、数据流与重算机制

### 8.1 局部状态原则

目录的所有状态都保留在 `ArticleView` 本地，不写入全局 store：

1. 目录项数据。
2. 当前激活章节。
3. 目录浮层是否显示。
4. 目录布局测量结果（位置、宽度、是否隐藏）。

### 8.2 重算触发时机

以下事件触发目录重新提取或重新布局：

1. 切换文章。
2. `bodyHtml` 变化。
3. 翻译视图变化导致正文结构变化。
4. 正文内容尺寸变化。
5. 阅读区尺寸变化。
6. 窗口尺寸变化。
7. 三栏拖拽导致右栏宽度变化。

### 8.3 推荐实现机制

1. 使用 `ResizeObserver` 观察正文内容根节点与阅读区容器。
2. 使用滚动事件只更新当前 active heading，不做高频全局同步。
3. 将布局计算封装为纯 helper，便于单测覆盖与后续调参。

## 9. 错误处理与边界

### 9.1 DOM 与数据边界

1. 如果正文根节点不存在或尚未渲染完成，目录直接隐藏。
2. 如果 heading 被翻译视图或内容刷新重建，应优先使用最新 DOM 重新查找目标节点。
3. 如果点击目录项时找不到目标 heading，则安全忽略，不抛出错误。

### 9.2 测量边界

1. 若测量得到的可用空白宽度小于隐藏阈值，目录直接隐藏。
2. 若计算结果出现非有限值，应回退到隐藏状态，避免异常样式闪动。
3. 若 `ResizeObserver` 回调频繁，应确保只在结果真实变化时更新局部 state。

### 9.3 可访问性边界

1. 目录容器使用语义化 `nav`，并提供明确的 `aria-label`。
2. 当前章节应有可感知的焦点/高亮反馈，而不是只靠颜色区分。
3. 键盘用户应可以顺序聚焦并点击目录项触发跳转。

## 10. 测试与验收设计

### 10.1 单元与组件测试

1. `ArticleView.outline.test.tsx`
   - 有 heading 且足够长时显示目录。
   - 短文章即使有 heading 也不显示目录。
   - 右侧空白不足时隐藏目录。
   - 点击目录项后滚动到对应标题。
2. `ArticleOutlineRail.test.tsx`（或新组件名对应测试）
   - 默认常驻可见。
   - 宽度收缩时仍保留截断与当前项高亮。
   - hover 只改变视觉强调，不改变显示/隐藏语义。
3. `articleOutline` helper 测试
   - 继续覆盖 heading 提取与稳定锚点。
   - 新增布局计算 helper 的边界测试。

### 10.2 验收标准

1. 桌面宽度下，长文章且存在 `h1/h2/h3` 时，目录常驻显示在正文右侧空白区。
2. 目录不改变正文实际阅读宽度。
3. 点击目录项后，正文平滑滚动到对应章节。
4. 正文滚动后目录仍停留在右上角，不会被正文带走。
5. 窗口缩放或三栏拖拽后，目录位置与宽度重新贴合新的右侧空白。
6. 右侧空白不足或文章长度过短时，目录自动隐藏。

## 11. 结论

本次重设计的核心，不是继续优化“目录轨道”的细节，而是把目录的角色重新定义为“正文外侧的低存在感辅助导航”。因此实现上应从“滚动内容内的 hover 控件”转向“阅读区内、正文外侧、可测量定位的常驻目录浮层”。这样既满足用户对点击跳转与滚动稳定性的要求，也尽量不干扰当前三栏布局与正文阅读宽度。
