# 设置日志模块设计

- 日期：2026-03-19
- 状态：已确认
- 需求：在“设置”中新增“日志”模块，支持开启/关闭系统日志记录，支持按预设保留天数清理日志；记录所有服务端外部 API 调用与系统级内部执行状态；在前端设置页原样展示日志，并支持按 `error / warning / info` 筛选。

## 背景

当前项目已经具备几个可以直接复用的基础：

- 设置中心位于 `src/features/settings/SettingsCenterDrawer.tsx`
  - 当前已有 `通用 / RSS / AI` 三个一级分区
  - 设置草稿、自动保存、校验与远端持久化已经成型
- 设置持久化位于 `src/store/settingsStore.ts` 与 `src/app/api/settings/route.ts`
  - 用户配置统一落在 `app_settings.ui_settings`
  - `settingsSchema` 已负责默认值与归一化
- 服务端存在稳定的异步执行链路
  - RSS 抓取由 `src/worker/index.ts` 驱动
  - AI 摘要、翻译、AI Digest 也都走 worker / queue / service 路径
- 当前已有错误透出增强设计
  - `docs/superpowers/specs/2026-03-19-async-error-surface-design.md`
  - 已经把“原始错误”保留为系统内的明确目标之一

但当前系统仍缺少统一日志能力：

- 没有独立的日志持久化模型
- 没有统一 logger 入口
- 设置页中没有日志查看面板
- 外部 API 请求与内部任务状态只零散散落在错误字段、数据库状态或 `console` 中

因此，本次设计的重点不是做一个完整 observability 平台，而是在现有设置中心与 worker 架构上，补一条“可开启、可持久化、可查看、可筛选”的系统日志主链路。

## 目标

- 在设置中心新增独立一级分区 `日志`
- 允许用户开启或关闭日志记录
- 允许用户从预设项中选择日志保留天数：`1 / 3 / 7 / 14 / 30 / 90`
- 将日志配置继续纳入现有设置持久化与自动保存链路
- 将日志正文持久化到独立数据库表，而不是塞进 `ui_settings`
- 记录服务端所有第三方出站 API 调用
- 记录系统级内部执行状态：RSS 抓取、AI 摘要、AI 翻译、AI Digest、设置保存等
- 在设置页中按时间倒序展示日志
- 支持按 `error / warning / info` 筛选
- 第三方错误响应若为 JSON，必须以字符串形式原样写入错误日志详情，并在前端原样展示

## 非目标

- 不把日志做成独立页面、命令面板或通知中心
- 不记录所有前端用户点击、切 tab、展开折叠等纯 UI 行为
- 不把浏览器到本服务的 `/api/*` 请求做成全量 HTTP 审计
- 不记录普通成功请求的请求体和响应体
- 不在第一版提供全文搜索、来源类型筛选、导出日志、单条删除、批量清空
- 不新增浏览器自动化测试
- 不引入新的外部日志平台或 SaaS

## 已确认输入

- 日志持久化范围：`C`
  - 日志落到服务端 / 数据库
  - 页面刷新、应用重启后仍可查看历史日志
- 数据采集策略：`C`，但附加约束
  - 常规第三方调用只记录元信息
  - 第三方错误响应必须保留原始返回内容
  - 若响应体是 JSON，必须将原始响应 JSON 以字符串形式写入错误日志详情
- 内部执行状态范围：`A`
  - 仅记录系统任务生命周期
  - 不记录所有前端用户操作
- 日志关闭行为：`A`
  - 关闭后停止新增
  - 历史日志继续保留并可查看
- 保留天数输入方式：`A`
  - 使用固定预设项：`1 / 3 / 7 / 14 / 30 / 90`
- 设置页布局选择：`A`
  - 日志作为设置中心的独立一级分区

## 备选方案

### 方案 A：把日志数组直接存进 `app_settings.ui_settings`

做法：

- 日志配置和日志正文都塞进现有设置 JSON
- 前端继续只走 `/api/settings`

优点：

- 表面改动最少

缺点：

- 日志是持续增长数据，不适合与用户配置混存
- 列表读取、清理、分页和筛选都会越来越别扭
- 很容易让 `ui_settings` 读写成本失控

### 方案 B：新增独立日志表，但由各处业务代码直接各写各的

做法：

- 新增日志表
- `apiClient`、worker、settings route、service 各自直接写 SQL 或 repository

优点：

- 比方案 A 更接近正确的数据模型

缺点：

- 字段格式会很快发散
- 不同链路写出的日志结构难以统一展示
- 后续维护成本高

### 方案 C：独立日志表 + 统一 logger 服务 + 设置页独立日志分区（推荐）

做法：

- 日志配置继续放进 `PersistedSettings`
- 日志正文写入独立 `system_logs` 表
- 所有日志都通过统一 `systemLogger` 服务写入
- 设置页新增 `日志` 一级分区，并通过专用 `/api/logs` 读取日志列表

优点：

- 配置与数据边界清晰
- 日志格式统一，便于前端原样展示
- 清理、分页、筛选天然合理
- 后续如果扩展来源类型或导出能力，不需要推翻第一版

缺点：

- 比前两种方案改动面更大

## 推荐方案

采用方案 C：独立日志表 + 统一 logger 服务 + 设置页独立日志分区。

理由：

- 与“持久化、原样展示、可筛选、可保留”的需求完全对齐
- 最大程度复用现有设置中心与 worker 架构
- 将增长型日志数据从 `ui_settings` 中剥离，避免未来返工

## 已确认设计

### 1. 总体架构

本次能力分成三层：

- 配置层
  - 扩展 `PersistedSettings`
  - 新增 `logging.enabled` 与 `logging.retentionDays`
- 存储层
  - 新增独立日志表 `system_logs`
  - 负责日志持久化、查询与过期清理
- 展示层
  - 设置中心新增 `日志` 一级分区
  - 通过专用日志接口读取列表并展示

明确边界：

- `/api/settings` 只负责日志配置
- `/api/logs` 只负责日志读取
- 不提供“前端直接写日志”的公开接口
- 所有日志统一由服务端内部调用 logger 服务写入

### 2. 设置模型与默认值

在 `PersistedSettings` 中新增：

- `logging.enabled: boolean`
- `logging.retentionDays: 1 | 3 | 7 | 14 | 30 | 90`

默认值：

- `logging.enabled = false`
- `logging.retentionDays = 7`

理由：

- 日志属于可选能力，默认关闭更稳妥
- `7` 天是一个兼顾可排障性与存储成本的合理默认值

落点：

- `src/types/index.ts`
- `src/features/settings/settingsSchema.ts`
- `src/store/settingsStore.ts`
- `src/app/api/settings/route.ts`
- 对应设置测试文件

### 3. 日志表与记录结构

新增表：`system_logs`

建议字段：

- `id bigint generated by default as identity primary key`
- `level text not null`
  - 仅允许 `error | warning | info`
- `category text not null`
  - 来源类型标签，例如 `external_api`、`settings`、`rss_fetch`、`ai_summary`、`ai_translate`、`ai_digest`
- `message text not null`
  - 简短摘要
- `details text null`
  - 原样错误详情或补充说明
- `source text not null`
  - 来源模块，例如 `worker/index`、`aiSummaryStreamWorker`、`settings route`
- `context_json jsonb not null default '{}'::jsonb`
  - 仅存非敏感元信息
- `created_at timestamptz not null default now()`

建议索引：

- `created_at desc`
- `(level, created_at desc)`

设计原则：

- `message` 用于列表摘要
- `details` 用于原样展示
- `context_json` 只承载可检索的非敏感结构化信息
- `category` 是内部来源标签，不作为第一版设置页的主筛选维度

命名规则需要在存储层与 API 层明确区分：

- 数据库列名固定为 `context_json`
- 对外 API 字段名统一为 `context`
- API route / DTO 负责把存储层的 `context_json` 映射为响应层的 `context`

### 4. 统一 logger 服务

新增服务层，例如：

- `src/server/logging/systemLogger.ts`

职责：

- 读取当前日志配置
- 在 `enabled = false` 时短路
- 统一规范日志字段
- 调用 repository 落库

建议统一调用形状：

- `writeSystemLog({ level, category, source, message, details, context })`

其中：

- `level`
  - `info`：正常开始、正常完成、正常状态推进
  - `warning`：重试、跳过、降级、部分失败但流程可继续
  - `error`：终止失败、第三方请求失败、内部异常
- `context`
  - 仅允许非敏感元信息，例如 `url`、`method`、`status`、`durationMs`、`feedId`、`articleId`、`provider`、`model`

明确约束：

- 业务层不直接写日志表
- 所有链路统一经由 logger 服务

### 5. 采集边界

“所有外部 API 调用”在本设计中收敛为：

- 服务端发往第三方的出站请求

包括：

- RSS 源抓取请求
- AI provider 请求
  - 摘要
  - 翻译
  - AI Digest
- 其他确实存在的服务端第三方请求

不包括：

- 浏览器到本服务的 `/api/*` 请求
- 纯前端 UI 交互

“内部执行状态”收敛为系统级任务生命周期：

- RSS 抓取任务
- AI 摘要任务
- AI 翻译任务
- AI Digest 任务
- 设置保存流程

不包括：

- 打开设置
- 切换 tab
- 点击筛选
- 纯展示态展开与折叠

### 6. 外部 API 日志规则

普通成功请求：

- 记录一条 `info`
- 仅记录元信息
- 不记录请求体与响应体

普通失败请求：

- 记录一条 `error`
- `message` 保存简短摘要，例如 `AI summary request failed`
- 存储层的 `context_json` 保存 `url / method / status / durationMs / provider / model` 等元信息

第三方错误响应规则是本次实现的硬约束：

- 如果第三方返回体是 JSON：
  - 必须把原始响应 JSON 以字符串形式写入 `details`
  - 不做中文翻译
  - 不做字段删改
  - 不做摘要提炼
- 如果调用链拿到了原始响应文本：
  - 优先直接保存原始文本
- 如果调用链只拿到了已解析对象：
  - 回退为 `JSON.stringify(parsedBody)`
- 如果返回体不是 JSON：
  - 保存原始文本到 `details`

### 7. 内部执行状态日志规则

记录范围：

- RSS 刷新：开始、成功、失败、重试
- AI 摘要：排队、开始、成功、失败
- AI 翻译：排队、开始、成功、失败、分段重试
- AI Digest：开始、成功、失败
- 设置保存：开始、成功、失败

等级规则：

- 成功完成：`info`
- 重试 / 跳过 / 降级：`warning`
- 终止失败：`error`

写入原则：

- 只记录关键生命周期
- 不把所有中间状态都扩成高噪声流水
- 同一动作的成功请求优先只记一条终态日志

### 8. 日志开关切换规则

需要把开关切换的边界行为写死，避免规划阶段歧义：

- 当日志当前为关闭，用户开启日志时：
  - 设置保存成功后，系统写入第一条 `info`
  - 内容可为 `Logging enabled`
- 当日志当前为开启，用户关闭日志时：
  - 系统写入最后一条 `info`
  - 内容可为 `Logging disabled`
  - 此后新的日志写入全部短路
- 当仅调整 `retentionDays` 时：
  - 若日志处于开启状态，记录一条 `info`
  - 若日志处于关闭状态，不补写配置变更日志

这意味着 logger 服务或 settings route 需要支持一次受控的“边界日志”写入，不能让启停本身被短路掉。

### 9. 日志清理与保留策略

日志过期清理由常驻 worker 驱动，而不是由前端页面触发。

做法：

- worker 周期性执行过期清理
- 删除 `created_at < now() - retentionDays`
- 清理频率建议为每小时一次

明确约束：

- 不在每次日志写入时顺手清理
- 不在前端打开“日志”分区时触发清理
- 调小保留天数后，旧日志可在下一次清理周期内被删除，不要求即时同步删除

理由：

- 当前系统已有常驻 worker 与周期性节拍
- 清理逻辑挂在后台更符合当前架构，也不会把查询与写入路径拖慢

### 10. API 设计

#### `/api/settings`

继续负责配置持久化。

新增返回与保存字段：

- `logging.enabled`
- `logging.retentionDays`

#### `/api/logs`

新增只读接口：

- `GET /api/logs`

建议查询参数：

- `level=error|warning|info`
- `limit`
- `before`

返回原则：

- 按 `created_at desc` 排序
- 返回日志项数组与下一页游标信息

分页协议在本设计中需要明确为“服务端生成的 opaque cursor”，避免前后端在时间戳或 ID 语义上各自解释：

- `before`
  - 表示“读取这条游标之前的更旧日志”
  - 它不是裸 `created_at`，也不是裸 `id`
  - 它必须是服务端返回的 opaque cursor，前端只负责透传，不自行解析
- `nextCursor`
  - 由服务端在本次响应中返回
  - 当前页还有更旧数据时返回非空字符串
  - 当前页已经到底时返回 `null`
- cursor 生成语义
  - 服务端基于当前页最后一条日志的 `(created_at, id)` 生成
  - 这样可以在相同时间戳下保持稳定翻页，不因只用时间戳而漏数或重数

建议响应形状：

- `items: SystemLogItem[]`
- `nextCursor: string | null`
- `hasMore: boolean`

其中日志项的对外字段名统一使用：

- `context`

它由后端从数据库列 `context_json` 映射而来；前端展示层只接触 `context`，不直接接触存储层命名。

示例：

请求：

- `GET /api/logs?level=error&limit=50`
- `GET /api/logs?level=error&limit=50&before=eyJjcmVhdGVkQXQiOiIyMDI2LTAzLTE5VDEwOjAwOjAwLjAwMFoiLCJpZCI6IjEyMyJ9`

响应：

```json
{
  "ok": true,
  "data": {
    "items": [
      {
        "id": "128",
        "level": "error",
        "category": "external_api",
        "message": "AI summary request failed",
        "details": "{\"error\":{\"message\":\"Rate limit exceeded\"}}",
        "source": "aiSummaryStreamWorker",
        "context": {
          "status": 429,
          "provider": "openai",
          "durationMs": 812
        },
        "createdAt": "2026-03-19T10:12:30.000Z"
      }
    ],
    "nextCursor": "eyJjcmVhdGVkQXQiOiIyMDI2LTAzLTE5VDEwOjEyOjMwLjAwMFoiLCJpZCI6IjEyOCJ9",
    "hasMore": true
  }
}
```

第一版不提供：

- 搜索参数
- category 筛选
- 手动删除接口

### 11. 前端设置页设计

设置中心新增第四个一级分区：

- `日志`

建议新增组件：

- `src/features/settings/panels/LogsSettingsPanel.tsx`

该面板分三块：

#### 配置卡片

- 日志开关
- 保留天数预设按钮或选择器：`1 / 3 / 7 / 14 / 30 / 90`
- 文案明确说明：
  - 关闭后停止新增
  - 历史日志继续保留并可查看

#### 筛选条

前端文案可以继续使用用户语义里的“分类”，但实际对应的是日志等级 `level`：

- `全部`
- `error`
- `warning`
- `info`

第一版只支持这一维筛选。

#### 日志列表

每条日志显示：

- 时间
- `level`
- `message`
- `category`
- `source`

若存在 `details`：

- 作为纯文本原样展示
- 若内容是 JSON 字符串，以等宽文本块展示

若存在 `context`：

- 挑选关键字段展示，例如 `status`、`url`、`durationMs`、`feedId`、`articleId`

展示要求：

- 按时间倒序
- 默认分页或游标加载
- 不将日志正文存入 `settingsStore`
- 由日志面板单独获取和管理展示态数据

### 12. 原样展示与安全边界

“原样展示”不等于允许任意内容直接作为 HTML 渲染。

前端必须遵守：

- `details` 一律按纯文本渲染
- JSON 字符串按代码块或等宽文本展示
- 不执行 HTML
- 不做富文本解析

服务端必须遵守：

- 常规成功日志不保存请求体与响应体
- `context_json` 仅保存非敏感字段

本轮唯一允许“原样进入详情”的高敏感面，是第三方错误响应文本；其余数据不升级为全量报文审计。

### 13. 模块拆分建议

推荐拆分为 5 个单元：

1. 日志配置层
   - `PersistedSettings`
   - `settingsSchema`
   - `settingsStore`
   - `/api/settings`
2. 日志存储层
   - migration
   - repository
3. 统一 logger 服务
   - 写入规范化
   - 开关短路
4. 业务接入层
   - RSS / AI / settings save
5. 前端日志面板
   - 设置页 UI
   - `/api/logs` 调用
   - 筛选与分页展示

这样可以保证配置、存储、写入、接入、展示五层彼此独立，后续写实施计划时也容易拆步骤。

## 风险与防护

主要风险：

- 统一 logger 漏接某些第三方请求，导致“所有外部 API 调用”只部分覆盖
- 不同业务模块写出的 `message / category / source` 风格不统一，影响可读性
- 关闭日志或调整开关时，如果边界日志处理不当，会出现“无法知道何时开始/停止记录”的断点
- 若日志列表不分页，历史数据增长后会拖慢设置页打开速度

防护策略：

- 强制所有日志经由统一 logger 服务
- 在 spec 对 `level / category / source / details / context_json` 职责做明确划分
- 为开关切换定义固定边界日志规则
- 第一版即采用分页或游标，而不是全量拉取
