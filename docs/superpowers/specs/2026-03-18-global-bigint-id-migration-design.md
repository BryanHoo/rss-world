# 全局 ID 从 UUID 迁移到 bigint identity 设计

## 1. 背景与问题

当前系统的业务实体 ID 使用 `uuid`（例如 `14480875-0437-4ec0-979e-475792cd25d7`），在调试、日志排查、人工沟通时可读性较差。  
用户希望使用更合理、更短的 ID 方案，并明确要求：

- 覆盖系统内所有持久化与接口相关 ID（不仅限于 `feeds/categories/articles`）
- 接受破坏性变更，可清库重建
- 接受公开 API 的资源 ID 可枚举
- API 与前端继续以 `string` 传输 ID

## 2. 目标与非目标

### 2.1 目标

- 将所有业务实体主键与关联外键从 `uuid` 统一改为 `bigint`
- 主键统一由 PostgreSQL 自动生成（`generated by default as identity`）
- 去除应用层对持久化主键的 `crypto.randomUUID()` 依赖
- API 层将 `uuid` 参数校验替换为数字字符串校验
- 保持前后端 TypeScript 类型为 `string`，避免 JS `number` 精度风险

### 2.2 非目标

- 不处理线上无损迁移（本次允许清库重建）
- 不改造前端纯临时 ID（如 toast UI 生命周期 ID）
- 不改变 API 路径结构（仅改变 ID 格式与校验）

## 3. 方案对比与最终决策

### 3.1 备选方案

1. 全量改为 `bigint identity`（最终采用）
2. `bigint + sequence`，应用层显式取号再写入
3. 保留 `uuid` 内部主键，新增对外短 ID

### 3.2 选择理由

采用方案 1，因为它在当前约束下最简单、最一致、维护成本最低：

- 不引入额外 ID 生成包
- 索引与存储更紧凑，查询与 join 成本更低
- 与“全系统统一改造 + 清库重建”目标一致

## 4. 数据库设计

## 4.1 设计原则

- 业务主键统一为 `bigint generated by default as identity`
- 所有关联外键同步改为 `bigint`
- 关联数组字段从 `uuid[]` 改为 `bigint[]`
- 不改动非业务键（例如 `schema_migrations.version`、`app_settings.id`）

## 4.2 表级改造清单

以下字段全部改造为 `bigint` 体系：

- `categories.id`
- `feeds.id`
- `feeds.category_id`
- `articles.id`
- `articles.feed_id`
- `article_tasks.id`
- `article_tasks.article_id`
- `article_translation_sessions.id`
- `article_translation_sessions.article_id`
- `article_translation_segments.id`
- `article_translation_segments.session_id`
- `article_translation_events.session_id`
- `article_ai_summary_sessions.id`
- `article_ai_summary_sessions.article_id`
- `article_ai_summary_sessions.superseded_by_session_id`
- `article_ai_summary_events.session_id`
- `ai_digest_configs.feed_id`
- `ai_digest_configs.selected_feed_ids`（`bigint[]`）
- `ai_digest_configs.selected_category_ids`（`bigint[]`）
- `ai_digest_runs.id`
- `ai_digest_runs.feed_id`
- `ai_digest_runs.article_id`
- `ai_digest_run_sources.run_id`
- `ai_digest_run_sources.source_article_id`

## 4.3 Migration 组织策略

- 直接改写现有 `0001~0020` migration 基线到 `bigint` 版本
- 删除业务表中的 `gen_random_uuid()` 依赖
- 保持现有索引与约束语义不变，仅替换 ID 类型
- 所有 migration 单测断言同步更新，确保 SQL 语义一致

## 5. 应用层设计

## 5.1 API 输入校验

- 将 `z.string().uuid()` 统一替换为数字字符串校验（例如 `/^[1-9]\d*$/`）
- 覆盖所有路径参数、body 中的 ID 字段和 ID 数组字段
- 影响范围：`feeds/categories/articles/ai-digests` 全部相关路由

## 5.2 Repository SQL 类型替换

- 全面替换：
  - `::uuid` -> `::bigint`
  - `::uuid[]` -> `::bigint[]`
  - `any($1::uuid[])` -> `any($1::bigint[])`
  - `unnest($1::uuid[])` -> `unnest($1::bigint[])`
- 删除 `coalesce($1::uuid, gen_random_uuid())` 等 UUID 生成表达式

## 5.3 服务层 ID 生成职责调整

- 由数据库负责新主键生成，服务层不再手动构造主键
- 重点路径：
  - `createAiDigestWithCategoryResolution`：移除 `feedId = crypto.randomUUID()`
  - `POST /api/articles/[id]/ai-summary`：session 创建时不再传 UUID

## 5.4 类型策略

- DB 层：`bigint`
- Node/TS 与 API：`string`
- 理由：`bigint` 在 PostgreSQL 驱动中默认可按字符串读取，避免 JS 数值精度问题

## 6. 关键数据流（改造后）

## 6.1 创建 AI Digest Feed

1. API 校验 `selectedFeedIds/categoryId` 为数字字符串格式
2. 服务层调用 repository 插入 `feeds`，不传 `id`
3. 数据库生成 `feeds.id`
4. repository `returning id` 返回字符串 ID
5. 后续配置写入 `ai_digest_configs.feed_id`

## 6.2 创建 AI Summary Session

1. API 校验 `articleId` 为数字字符串
2. `upsertAiSummarySession` 插入 session 时不传显式主键
3. 数据库生成 `article_ai_summary_sessions.id`
4. 队列任务与事件流使用该字符串 ID 继续传递

## 6.3 分类重排与批量过滤

- `reorderCategories` 使用 `bigint[] + unnest`
- `ai_digest` 目标源过滤使用 `any($1::bigint[])`
- 批量插入 run sources 使用 `::bigint`

## 7. 错误处理与兼容策略

- 非数字字符串 ID：返回 `ValidationError`，保持现有错误包络结构
- 外键不存在：保持现有 `23503` 映射策略
- 由于是清库重建，不提供 UUID 兼容兜底与双读双写

## 8. 测试与验证设计

## 8.1 Migration 测试

- 更新 migration SQL 断言：
  - `uuid/gen_random_uuid/uuid[]` -> `bigint/identity/bigint[]`
- 覆盖关键表：`article_tasks`、翻译会话、摘要流、AI digest 配置与来源映射

## 8.2 API 路由测试

- fixture ID 从 UUID 改为数字字符串（例如 `1001`）
- 参数非法用例改为非数字字符串（如 `abc`、`-1`、`1.2`）
- 保持成功/失败状态码与错误响应结构稳定

## 8.3 Repository 与集成测试

- 覆盖 `::bigint[]` 查询路径与批量写入路径
- 验证 `insert ... returning id` 全链路可用
- 验证与 `ai_digest` 相关的多表关联查询不回归

## 8.4 构建验证

- 必须执行 `pnpm build` 作为完成标准
- 推荐执行 `pnpm test:unit` 作为回归补充

## 9. 风险与约束

- 风险：ID 可枚举（已确认接受）
- 风险：改造面覆盖所有 API 与 repository，漏改会导致运行时类型错误
- 约束：当前工作区存在其他未提交改动，本次仅提交 spec 文档，不触碰其余文件

## 10. 验收标准

- 代码库持久化链路不再出现业务 ID 的 `uuid` 定义与 `gen_random_uuid()` 生成
- API 不再使用 `z.string().uuid()` 校验业务 ID
- 所有 ID 相关测试与 migration 断言通过
- `pnpm build` 成功

## 11. 实施边界总结

本设计只定义“全局 ID 体系迁移到 `bigint identity`”的方案与验收口径。  
具体代码改造步骤将在后续 implementation plan 中拆解执行。
