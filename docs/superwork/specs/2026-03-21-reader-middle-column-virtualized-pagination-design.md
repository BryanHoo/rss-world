# 中栏文章列表按需加载与虚拟化设计

## 背景

当前阅读页中栏通过 `/api/reader/snapshot` 一次读取文章快照。服务端 `readerSnapshotService` 默认只返回 50 篇文章，前端 `ArticleList` 则把当前 `articleSections` 全量渲染到 DOM。

这导致两个问题同时存在：

- 当前视图最多只能看到前 50 篇，无法继续向下读取更旧文章
- 如果简单放开限制并继续全量渲染，随着文章数量增长，中栏 DOM 会快速膨胀，滚动、重排和图片观察都会变卡

本次需求明确约束如下：

- 中栏需要支持加载当前视图下的全部文章
- 文章应采用按需滚动加载，而不是首次全量拉取
- 已加载过的数据可以保留在内存中
- 离开视口较远的 DOM 节点允许卸载，只保留滚动占位
- 切换视图后重置分页和滚动会话，不恢复上个视图的加载位置
- 中栏右上角计数继续显示当前视图总篇数
- 当顶部插入新文章时，保持当前视口稳定，不自动跳到顶部
- `card` 和 `list` 两种模式都要支持按需加载和虚拟化

## 目标

- 让中栏可以继续向下读取当前视图的所有文章
- 将中栏渲染成本控制在稳定范围内，避免随着加载页数增长而明显卡顿
- 保持当前阅读流畅性，尤其是未读过滤、文章选择和顶部刷新体验

## 非目标

- 不做跨视图的分页位置恢复
- 不新增用户可配置的分页大小或虚拟化参数
- 不在本次需求中引入浏览器端可视化性能面板或额外诊断页面
- 不改造右栏文章内容区域的渲染策略

## 方案对比

### 方案 A：服务端分页 + 前端轻量虚拟化

服务端继续提供基于 `cursor` 的文章分页，前端持有当前视图已加载的数据，并将 section 标题和文章条目拍平成虚拟行，只渲染视口附近的节点。

优点：

- 直接解决“只能看到 50 篇”和“DOM 过多卡顿”两个核心问题
- 可以沿用现有 `/api/reader/snapshot`、`cursor`、`nextCursor` 设计
- 不需要引入第三方虚拟化依赖，便于贴合当前 section 分组结构
- 更容易做“顶部插入时保持视口稳定”的滚动补偿

缺点：

- 需要自行维护虚拟窗口、滚动补偿和底部加载触发
- 要调整当前 `ArticleList` 对全量 DOM 的若干假设

### 方案 B：服务端分页 + 引入通用虚拟化库

分页策略不变，前端使用现成虚拟化库管理可见窗口和 overscan。

优点：

- 虚拟滚动基础能力更成熟
- 如果以后出现动态高度需求，扩展路径更清晰

缺点：

- 需要新增依赖并适配当前按天分组、双展示模式的结构
- 当前行高相对稳定，引库收益不明显
- 改造成本高于本次需求实际需要

### 方案 C：仅做分页加载，不做虚拟化

滚动到底继续加载更多数据，但已渲染节点持续累积在 DOM 中。

优点：

- 实现最快

缺点：

- 只能缓解首次加载问题，无法解决越滚越卡
- 与需求中“要考虑之前 DOM 特别多导致卡顿”的重点不匹配

## 结论

采用方案 A：服务端游标分页 + 前端轻量虚拟化。

原因是当前文章列表的 section 标题、`list` 行和 `card` 卡片高度都足够稳定，可以用固定高度模型实现较低复杂度的虚拟列表，同时满足现有交互约束。

## 架构设计

本次变更拆成三个层面：

1. Snapshot API 从“默认取前 50 篇”调整为“支持无限滚动的数据源”
2. Store 从“单次整页替换”调整为“当前视图的分页会话管理”
3. `ArticleList` 从“全量 section 渲染”调整为“扁平虚拟行 + 底部按需加载”

整体数据流如下：

1. 进入某个视图后，请求第一页快照
2. 服务端返回 `items`、`nextCursor`、`totalCount`
3. store 记录当前视图的已加载文章和分页状态
4. `ArticleList` 基于已加载文章构建 section 与虚拟行
5. 用户滚动接近底部时，前端用 `nextCursor` 拉取下一页并追加到当前会话
6. 页面只渲染视口附近的 section 标题和文章行
7. 刷新第一页且顶部出现新文章时，列表执行滚动补偿，保持当前视口稳定

## 服务端设计

### `/api/reader/snapshot`

保留现有接口，不新增独立分页接口，但扩展其语义：

- 保留 `view`
- 保留 `limit`
- 保留 `cursor`
- 新增 `unreadOnly`
- 保留 `includeFiltered`

返回结构扩展为：

- `articles.items`
- `articles.nextCursor`
- `articles.totalCount`

其中：

- `items` 为当前页文章
- `nextCursor` 为下一页游标，没有下一页时为 `null`
- `totalCount` 为当前查询条件下的文章总数，用于中栏顶部总计数展示

### 查询与过滤

服务端查询条件需要同时考虑：

- 当前视图 `view`
- 仅未读筛选 `unreadOnly`
- 特定 feed 视图下的 `includeFiltered`
- 游标翻页条件

服务端仍保持按 `(published_at, id)` 倒序分页，避免分页结果在时间相同文章上不稳定。

### limit 策略

当前服务端默认 `limit=50`，最大 `MAX_LIMIT=200`。本次设计不要求去掉上限，而是将其作为单页大小约束保留。

推荐策略：

- 首屏使用稳定分页大小，例如 50
- 每次继续滚动仍按固定页大小拉取下一页
- 不允许客户端一次请求“全部文章”

这样可以把负担稳定分摊到分页请求上，避免服务端和前端同时被一次大查询拖慢。

## Store 设计

当前 `loadSnapshot()` 会直接用本次结果替换某个视图的文章数组。本次需要引入“当前视图会话态”，但仍保持“切换视图即重置”的约束。

建议新增当前视图分页状态字段，至少包括：

- `articles`
- `nextCursor`
- `hasMore`
- `totalCount`
- `initialLoading`
- `loadingMore`
- `loadMoreError`
- `requestId`

说明如下：

- `articles`：当前视图已加载文章
- `nextCursor`：下一页游标
- `hasMore`：是否还有下一页
- `totalCount`：当前视图总数
- `initialLoading`：第一页加载中
- `loadingMore`：追加页加载中
- `loadMoreError`：追加页失败时的错误态
- `requestId`：用于丢弃过期请求

### 视图切换

用户已明确选择“切换视图就重置”。因此：

- 切换 `selectedView` 时清空当前分页状态
- 滚动位置回到顶部
- 重新请求第一页
- 不恢复上个视图的 `cursor`、已加载页或滚动锚点

### 请求模式

store 层拆分为两类加载动作更清晰：

- `loadSnapshot`：加载当前视图第一页
- `loadMoreSnapshot`：基于 `nextCursor` 追加下一页

两者都要做过期保护：

- 发请求时递增 `requestId`
- 响应回来后，如果视图已切换或 `requestId` 已过期，则直接丢弃

### 与现有文章详情缓存的关系

当前 `mergeSnapshotArticleWithExistingDetails()` 会保留已有文章的全文、AI 摘要、翻译等详情字段。这个策略要继续保留。

这样即便分页追加或第一页刷新替换，已打开过文章的详情仍能与新的列表快照合并，避免阅读态倒退。

## 中栏虚拟化设计

### 扁平虚拟行

当前 `articleSections` 适合展示，但不适合直接做虚拟窗口。建议在派生层增加“扁平行模型”，行类型仅保留：

- `section`
- `article`

每一行都具备：

- `key`
- `type`
- `height`
- `sectionKey`
- 文章或标题对应的数据引用

`ArticleList` 根据当前展示模式选择固定高度：

- section 标题高度固定
- `list` 文章行高度固定
- `card` 文章卡片高度固定

这样可通过累计高度直接推算滚动总高，而不依赖运行时全量测量。

### 虚拟窗口

组件维护：

- 当前容器 `scrollTop`
- 容器高度
- 当前渲染区间
- `overscan`

渲染时：

- 顶部 spacer 占位不可见行的累计高度
- 中间仅渲染窗口内真实节点
- 底部 spacer 占位剩余高度

这样无论已加载 50 篇还是 500 篇，DOM 数量都接近一个稳定上限。

### 底部继续加载

由于虚拟化后最后一项未必常驻 DOM，本次不依赖“观察最后一个元素”来触发翻页。推荐直接基于滚动位置判断：

- 当 `scrollHeight - (scrollTop + clientHeight)` 小于阈值时触发 `loadMoreSnapshot`
- 如果 `loadingMore=true`，则不重复请求
- 如果 `hasMore=false`，则停止触发

这种方式与虚拟窗口更一致，也避免由于最后一项被卸载导致观察器失效。

## 滚动稳定性设计

### 顶部插入补偿

用户明确要求：刷新后顶部插入新文章时，当前视口不跳动。

实现策略：

1. 刷新第一页前记录当前锚点行及其相对容器顶部偏移
2. 第一页新数据返回后，重新计算锚点行在新行模型中的理论位置
3. 用差值补偿 `scrollTop`

这样即使顶部新增多行，用户眼前原本正在看的内容仍停在接近原来的位置。

### 视图切换与重置

切换到别的视图不做补偿，直接：

- 清空当前虚拟窗口状态
- 重置滚动到顶部
- 加载新视图第一页

这与用户选择的“切走就重置”一致，也能避免复杂的跨视图滚动恢复逻辑。

## 与现有功能的兼容

### 未读过滤

当前 `showUnreadOnly` 下会通过 `retainedVisibleArticleIds` 保留已出现在列表中的文章，避免用户读完当前项后列表突然收缩。

本次保留这一行为，但作用边界改为“当前已加载文章集合”：

- 已加载范围内继续保持阅读连续性
- 未加载的旧文章不因为 retention 逻辑被提前引入

这样既保留现有体验，也不会破坏分页边界。

### 预览图懒加载

当前预览图通过 `IntersectionObserver` 观察卡片节点。本次仍保留懒加载，但观察范围只覆盖当前渲染窗口里的节点。

这样可以减少：

- 观察器绑定数量
- 无意义的图片预热
- 已离屏节点对应的图片状态维护开销

### 标题换行测量

当前 `card` 模式通过遍历 `cardTitleRefs` 测量标题是否换行，再决定摘要显示一行还是两行。这依赖全量 DOM 常驻，不适合虚拟化场景。

本次建议优先简化为稳定固定布局，不再依赖全量测量。如果保留该视觉细节，也必须改为“仅对可见节点测量”，不能再建立对全量节点的隐含假设。

## 错误处理

### 第一页加载失败

- 保持空态或错误态
- 不进入部分渲染状态
- 允许用户通过刷新动作重新触发

### 追加页加载失败

- 保留已加载内容
- 在列表底部显示轻量失败提示
- 提供重试加载入口
- 不清空现有 `nextCursor` 之前的可读内容

### 过期响应

以下情况直接丢弃响应：

- 当前视图已切换
- 同一视图发起了更新的请求

避免旧分页结果覆盖新视图或新刷新结果。

## 测试策略

至少覆盖以下几类测试：

### 1. 服务端分页与计数

- 第一页返回 `items`、`nextCursor`、`totalCount`
- 传入 `cursor` 时返回下一页
- `unreadOnly` 能正确收窄结果和总数

### 2. Store 分页状态

- 首次加载写入第一页文章和 `nextCursor`
- 继续加载会追加文章，而不是覆盖
- 视图切换后分页状态重置
- 过期请求不会覆盖当前状态
- 追加失败不会清空已有文章

### 3. 中栏虚拟化

- 初始只渲染窗口附近行，而非全部 DOM
- 滚动接近底部时触发下一页加载
- `card` / `list` 两种模式都能正常工作
- 顶部插入新文章时，当前视口保持稳定

### 4. 构建验证

- 所有代码修改后必须运行 `pnpm build`

## 风险与约束

- 虚拟化会打破当前若干依赖全量 DOM 的隐式逻辑，需要逐项收敛
- 如果 `card` 和 `list` 的真实高度与固定高度偏差过大，滚动位置可能出现轻微误差，需要在实现时校准常量
- 聚合视图在文章很多时，`totalCount` 查询成本需要关注，但它仍优于一次性回传全部文章

## 验收标准

- 中栏不再只能展示前 50 篇，滚动到底可继续读取更旧文章
- 页面不会因已加载文章增多而持续累积大量 DOM
- 中栏右上角继续显示当前视图总篇数
- 切换视图后分页和滚动状态重置
- 顶部刷新插入新文章时，当前视口内容保持稳定
- `card` 和 `list` 两种模式都支持按需加载与虚拟化
